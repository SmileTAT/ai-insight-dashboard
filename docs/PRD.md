## 一、项目定位

**ai-insight-dashboard** 是一个战略级 AI 情报系统，通过自动化数据采集与 AI 分析，聚焦**技术路线图演进**与**大厂竞争格局**两大维度，以定期（周报/月报）深度报告为核心产出，为 AI 从业者与研究者提供结构化的行业决策参考。

> 基于 agents-radar 二次开发，继承其自动化采集框架（TypeScript + GitHub Actions），重构分析与报告模块以匹配战略级情报需求。

### 与 agents-radar 的差异定位

| 维度 | agents-radar | ai-insight-dashboard |
| :--- | :--- | :--- |
| 产出节奏 | 日报为主 | 周报 + 月报（不做日报） |
| 内容视角 | 信息聚合（signals digest） | 战略分析（技术演进 + 竞争格局） |
| 分析深度 | 分类汇总 | 赛道归类、前序对比、趋势判断 |
| 数据源数量 | 10 个 | MVP 收敛为 3 类核心源 |

## 二、核心目标

1. **追踪技术迭代路径**：识别关键技术的演进脉络，回答"这条技术线最近发生了什么变化？相较于之前有哪些突破？"
2. **透视大厂战略布局**：监控头部玩家的集中动作，回答"这家公司最近在押注什么方向？竞争态势如何变化？"
3. **产出结构化深度报告**：以周报和月报为载体，将碎片信息整合为可阅读、可存档、可分享的战略内参。

## 三、用户画像

| 维度 | 描述 |
| :--- | :--- |
| 主要用户 | AI 技术决策者、技术战略研究员、AI 产品经理、VC 投资人 |
| 核心诉求 | 用最短时间掌握行业关键动向，辅助决策 |
| 使用场景 | 每周一早晨阅读周报；每月初复盘上月技术/竞争格局变化 |

## 四、MVP 功能范围

### P0（必须要有，MVP 核心）

| 模块 | 功能 | 说明 |
| :--- | :--- | :--- |
| **数据采集** | arXiv 论文抓取 | 按 `cs.AI`、`cs.LG`、`cs.CL` 类别，通过官方 API 采集近 7 天论文元数据；**必须经过两级漏斗过滤**（见 5.2），控制进入 LLM 分析的数量上限 |
| | GitHub 大厂组织监控 | 监控目标组织的新仓库与 Release，组织清单见 4.1 |
| | 官方博客监控 | 通过 RSS 或 Sitemap 差分追踪 OpenAI、Google DeepMind、Anthropic 的官方博客更新（Anthropic 无官方 RSS，沿用 agents-radar 的 sitemap 检测方案） |
| **数据持久化** | 原始数据归档 | 每次采集结果以 JSON 归档至 `data/YYYY/week-WW/`，**作为月报聚合的数据基础**（月报不重新采集，直接汇总当月各周数据） |
| **AI 分析** | 技术迭代摘要 | 识别每条信息所在的技术赛道，提炼相较前序工作的改进点（依据摘要中的自述对比 + 历史标签库，见 5.3） |
| | 大厂动作归类 | 将每条信息归入赛道标签（枚举见 5.1），并标记对应公司 |
| **报告生成** | 周报（Markdown） | 固定结构：技术迭代亮点 + 大厂动态汇总 + 趋势一句话总结 |
| | 月报（Markdown） | 深度结构：本月技术演进图谱 + 大厂竞争格局对比 + 下月看点预判 |
| **自动化** | GitHub Actions 定时触发 | 每周一 00:00 UTC（北京时间 08:00）生成周报；每月 1 日 00:00 UTC 生成月报；均支持 `workflow_dispatch` 手动触发以便调试与补跑 |
| **告警** | 失败自动开 Issue | 任一采集/分析/生成 Job 失败时，自动在仓库创建带 `pipeline-failure` 标签的 Issue，内含失败 Job 与日志链接 |
| **交付** | 报告提交至仓库 | 生成的报告自动 commit 到 `reports/weekly/` 与 `reports/monthly/` |
| | 网页版（v1.2 新增） | 同步生成静态站到 `docs/`（单期页面 + 首页目录，内联 CSS 零外部依赖，支持暗色模式），经 GitHub Pages（main 分支 `/docs`）对外发布 |

## 五、数据模型与分析规则

### 5.1 统一数据结构（MVP 简化版）

每条原始信息经采集后统一抽象为以下结构。**字段键名一律使用英文**（便于代码处理与跨模型提示词稳定性），中文仅出现在值与报告呈现层：

```json
{
  "id": "uuid",
  "source": "arxiv | github | blog",
  "company": "openai | google | anthropic | meta | microsoft | other",
  "publish_date": "2026-07-13",
  "title": "原文标题",
  "url": "原文链接",
  "raw_content": "摘要或描述文本",
  "ai_tags": {
    "track": "foundation-model",
    "directions": ["model-arch", "on-device"],
    "keywords": ["MoE", "RLHF"],
    "relevance": 4,
    "title_zh": "MoE 架构扩展至万亿参数"
  }
}
```

标签体系为**两层结构**（v1.1 修订）：

- `track` 为**一级赛道，单选稳定枚举**：`foundation-model`（基础模型）、`multimodal`（多模态）、`agent`（Agent）、`reasoning`（推理）、`alignment`（对齐/安全）、`application`（应用/产品）、`infra`（训练与推理基础设施）、`other`。稳定坐标系专用于月报的跨周期演进对比。
- `directions` 为**二级研究方向，受控词表多选（0-2 个）**：如 `gui-agent`、`on-device`、`memory`、`code-agent`、`world-model`、`rl-post-training` 等，词表在 `src/config.ts` 维护、可随热点增删。周报按方向聚类展示；用户可通过 `FOCUS_DIRECTIONS` 配置关注方向（默认聚焦 Agent 全部细分方向 + 记忆），命中方向在周报中置顶并标记 🎯。
- `relevance` 为**战略相关性 1-5 分**（LLM 评定）：5=重大发布/方向性突破，1=客户案例/公关文/人事通稿。低于 `MIN_REPORT_RELEVANCE`（默认 3）的条目**只归档不进报告**，从数据层拦截信息噪音。
- `title_zh` 为 LLM 生成的中文短标题（≤20 字），报告展示用；原文标题保留在链接中。
- `keywords` 为开放式技术关键词列表（LLM 抽取），用于跨周期的演进脉络匹配。
- 去重规则：以 `url` 规范化后为唯一键；arXiv 以论文 ID 为键（同一论文多版本只保留最新）。

### 5.2 arXiv 两级漏斗（成本与信噪比控制）

`cs.AI + cs.LG + cs.CL` 每周新增论文量在数千篇量级，不能全量送入 LLM：

1. **规则粗筛（零成本）**：命中任一条件进入候选池 —— ① 作者机构含目标大厂/知名实验室；② 标题/摘要命中关键词库（可配置）；③ 属于综述/基准类高信号论文。
2. **LLM 精筛（限额）**：候选池按信号分排序，每周最多 **N=50** 篇进入完整分析（N 可配置），其余仅保留元数据归档。

### 5.3 "改进点提炼"的可行性边界

LLM 仅能看到单篇摘要，无法真正检索前序工作。MVP 阶段的对比依据为：① 摘要中作者自述的 baseline 对比；② 本系统历史归档中同 `keywords` 条目的时间线。提示词须明确要求"仅基于给定材料，无对比依据时输出'首次追踪，暂无前序对比'"，**禁止编造对比结论**。

## 六、报告输出规格

### 周报结构（v1.2 修订：结论式标题 + 三段头条 + 编辑人格，阅读时长 < 5 分钟）

文案规范（对标 The Batch / Import AI）：
- 所有条目标题为**结论式 claim**（含关键数字，如"KV cache 复用让过程奖励模型提速 5000 倍"），禁止名词式论文题翻译
- 条目行展示**人话版 takeaway**（≤40 字、≤1 个术语），专业版改进点保留在头条与归档
- **编辑观察**栏目为犀利有观点的第一人称短评（哪个方向被高估/低估、大厂意图），判断须可由本周条目推导，禁止编造

```markdown
# AI 情报周报 | YYYY-MM-DD

> 第 N 期 ｜ 预计阅读 X 分钟

## 📌 本周要点
- [3 条，第 1 条为本周最大反差/惊喜，每条尽量带数字钩子]

## 💬 编辑观察
> [100-150 字观点短评]

## 🎯 本周头条
### [结论式标题]（1-3 条，LLM 编辑级筛选）
**发生了什么**：[2 句内的事实与规模]
**怎么做到的**：[1 句技术路径，可省略]
**为什么重要**：[对格局/技术路线的增量判断，禁复述上文]

## 🔬 研究方向雷达（仅展示本周有信号的方向，关注方向置顶标 🎯）
### 📡 [方向名]（本周 N 条）🎯
> [方向小结：发生了什么、往哪演进]
- 🔥/⭐ **[中文短标题]**（来源/公司）— [改进点提炼] [原文](url)

## 🏢 大厂动态
| 公司 | 本周战略信号 | 关键动作 |
|------|-------------|----------|
| OpenAI | [一句话战略判断] | [精选 2-3 项] |

## 🔭 下周关注
[1-2 个值得提前关注的节点]
```

信号分级：🔥 = relevance 5（重大），⭐ = relevance 4（显著）。头条条目不在雷达区重复出现。

### 月报结构（约 1500-2500 字，阅读时长 < 15 分钟）

```markdown
# AI 情报月报 | YYYY-MM

## 📌 本月综述
[宏观趋势判断，200-300 字]

## 🗺️ 技术演进图谱
### 赛道一：[基础模型]
- 演进节点 1（月初）: ...
- 演进节点 2（月中）: ...
- 演进节点 3（月末）: ...
→ 本月该赛道核心方向：[...]

### 赛道二：[多模态]
...

## 🏢 大厂竞争格局月报
| 公司 | 本月关键发布 | 战略重心判断 | 生态动作 |
|------|-------------|-------------|----------|
| OpenAI | ... | ... | ... |

## 📊 竞争格局小结
[本月谁在进攻、谁在防守、哪些赛道在升温]

## 🔭 下月看点预判
1. ...
2. ...
```

月报数据来源：直接聚合 `data/YYYY/week-WW/` 中当月各周归档 + 各周周报结论，不重新采集。

## 七、技术选型（MVP）

| 组件 | 选型 | 说明 |
| :--- | :--- | :--- |
| 语言与采集框架 | **TypeScript（继承 agents-radar）** | agents-radar 本体为 TypeScript，"复用其采集模块"是本项目立项前提，故不引入 Python 重写；arXiv/GitHub/sitemap 采集器均有现成实现可裁剪 |
| 自动化流水线 | GitHub Actions | 继承 agents-radar，免费、托管；cron 以 UTC 表达 |
| AI 分析 | 可配置的 LLM Provider 抽象层 | 默认低成本模型跑周报与批量归类；月报综述允许切换更强模型。模型名与 API 端点走环境变量配置，不硬编码 |
| 密钥管理 | GitHub Actions Secrets | LLM API Key 仅存于 repo Secrets，禁止入库 |
| 输出格式 | Markdown | 直接入库，便于版本管理和 Pages 展示 |
| 状态与归档 | JSON 文件（入库） | `state.json` 记录各源上次采集游标；`data/` 归档原始数据支撑月报聚合。SQLite 留待 P1 |

## 八、MVP 迭代路线图

| 阶段 | 时间 | 交付物 |
| :--- | :--- | :--- |
| **Phase 0：脚手架** | Day 1-2 | Fork/裁剪 agents-radar，删除不需要的 7 个数据源，跑通基础工作流，能生成一份 Hello World 报告 |
| **Phase 1：数据源重构** | Day 3-5 | 按 4.1 清单配置数据源：arXiv 两级漏斗 + 大厂 GitHub 组织 + 3 个官方博客，验证采集稳定性与去重 |
| **Phase 2：分析模块开发** | Day 6-10 | 赛道归类提示词 + 改进点提炼提示词（含 few-shot 与"禁止编造"约束），人工抽检调优 |
| **Phase 3：报告模板落地** | Day 11-14 | 周报/月报 Markdown 生成器 + 数据归档与月报聚合逻辑 + 失败告警 Issue |
| **Phase 4：自动化上线** | Day 15 | 配置定时任务与 `workflow_dispatch`，手动触发验证全链路 |
| **Phase 5：内测与调优** | Day 16-29（2 周观察期） | 无人值守连续运行 2 周，产出 2 期周报 + 1 期月报，人工校验质量，迭代提示词 |

> 注：Phase 5 的"连续运行 2 周"是自然时间，不可压缩，故整体周期为约 4 周而非 3 周。

## 九、MVP 成功标准

- [ ] 系统可**无人值守连续运行 2 周**，无未处理的报错中断（允许单次失败但需触发告警 Issue 且可通过手动补跑恢复）
- [ ] 每周一北京时间 **08:00-10:00 窗口内**产出当周周报，每月 1 日同窗口产出月报（GitHub Actions cron 不保证准点，高峰期可延迟数十分钟，故以窗口为准）
- [ ] 周报中**技术迭代亮点**的"改进点提炼"人工抽检 10 条中 ≥ 7 条被确认有参考价值，且无编造的对比结论
- [ ] 月报中的**大厂竞争格局对比**覆盖至少 4 家公司（OpenAI、Google、Anthropic、Meta）
- [ ] 报告阅读时长：周报 < 5 分钟，月报 < 15 分钟
- [ ] 单月 LLM API 成本 ≤ 预算上限（立项时确定具体数值，由 5.2 的分析限额 N 反向控制）

## 十、风险与应对

| 风险 | 影响 | 应对 |
| :--- | :--- | :--- |
| LLM 输出质量不稳定 / 编造对比 | 报告价值降低甚至误导决策 | 精细提示词 + few-shot 示例；无依据时强制输出"暂无前序对比"；成功标准中设人工抽检环节 |
| arXiv 信息量过大 | 成本失控、信噪比低 | 两级漏斗（规则粗筛 + LLM 限额精筛），限额 N 可配置 |
| API 调用成本超预期 | 不可持续 | Provider 抽象层默认低成本模型；每次运行记录 token 消耗并写入运行日志；月报才启用强模型 |
| 数据源反爬/限流 | 采集失败 | arXiv/GitHub 优先官方 API；sitemap 差分低频访问；重试机制 + 失败自动开 Issue |
| GitHub Actions cron 延迟或超时 | 报告延迟/未生成 | 成功标准以时间窗口计；采集与分析拆分为多个 Job；所有工作流支持 `workflow_dispatch` 手动补跑 |
| Anthropic 等无 RSS 源结构变化 | 博客监控静默失效 | sitemap 差分 + "连续 N 天无新增"心跳检测，异常时开 Issue 提示人工确认 |

## 十一、非目标（Out of Scope）

- 不做日报（与 agents-radar 的核心差异点）
- 不做实时推送与订阅分发（P1）
- 不做二级市场/融资情报（信息源与合规成本高，与技术路线图定位不符）
- 不承诺分析结论的投资建议属性，报告仅作研究参考
